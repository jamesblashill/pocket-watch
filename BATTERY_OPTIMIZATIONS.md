# Battery-Life Optimization Findings

Source: battery-life review of the alarm clock app (2026-07-15). Status column is meant to be
updated as items are picked up — use `Not started` / `In progress` / `Done` / `Wontfix`.

| # | Priority | Issue | Battery impact | Difficulty | Affected files | Recommended change | Side effects | Status |
|---|---|---|---|---|---|---|---|---|
| 1 | Critical | Automatic light sleep (screen-off state) is capped at ~5 ms per nap because the LVGL adapter's tick timer (`tick_period_ms=5`) keeps firing an `esp_timer` even while the worker is paused | High — likely the single biggest lever; today's "sleep" is thousands of 5 ms micro-naps/sec, not sustained sleep | Moderate (needs adapter change or LVGL tick-callback approach) | `components/waveshare__esp32_s3_touch_lcd_1_85B/esp32_s3_touch_lcd_1_85B.c:1088-1091`, `components/esp_lvgl_adapter/src/adapter/esp_lv_adapter.c` | Switch LVGL to `lv_tick_set_cb(esp_timer_get_time)`-style on-demand ticking (no periodic timer needed) if the adapter supports it, or stop/lengthen the tick timer while paused and reconcile `lv_tick_inc()` on resume | Requires patching or configuring a managed component; must verify LVGL's animation/timer math tolerates a large single tick jump on wake | Done — forked `espressif/esp_lvgl_adapter` into `components/esp_lvgl_adapter` (pinned via `override_path` in the board's `idf_component.yml`) and switched `tick_init()` to `lv_tick_set_cb()` backed by `esp_timer_get_time()`; periodic tick esp_timer removed entirely, so nothing caps the light-sleep window anymore. Needs on-device battery/current measurement to confirm the win; item #2 (GPIO wake sources) still required to unlock long sleep windows in practice. |
| 2 | Critical | No GPIO wake sources are registered for light sleep (no `gpio_wakeup_enable`/`esp_sleep_enable_gpio_wakeup`/`ext0`/`ext1` anywhere) | High — wake currently depends entirely on periodic timers, not real touch/button interrupts; fixing this unlocks much longer sleep windows once the tick-timer issue above is also fixed | Low-moderate | New code near `main/app_clock/lvgl_screen_power_button.c:103`, touch INT = `GPIO_NUM_4`, power button = `GPIO_NUM_0` | Call `gpio_wakeup_enable(GPIO_NUM_4/0, GPIO_INTR_LOW_LEVEL)` + `esp_sleep_enable_gpio_wakeup()` when entering the screen-off state | Must confirm GPIO4/GPIO0 are valid as light-sleep wake sources on ESP32-S3 (they are — light-sleep GPIO wake works on any digital GPIO, unlike deep-sleep `ext0/ext1` which need RTC IOs) | Done — power button (GPIO0) uses `iot_button`'s native `enable_power_save` flag on `button_gpio_config_t`, which arms/disarms `gpio_wakeup_enable` + `esp_sleep_enable_gpio_wakeup()` automatically as the button driver's own idle state machine allows. Touch INT (GPIO4) is armed/disarmed by hand in `screen_power_set_on()` since `esp_lcd_touch` has no equivalent facility - safe because the touch driver never registers a GPIO ISR on that pin (LVGL polls `get_xy()` instead), so repurposing its interrupt config for wake doesn't fight anything. Built and tested on-device. |
| 3 | High | LCD controller itself is never put to sleep or even `DISPOFF`'d — only the LED backlight duty is set to 0 | Medium — ST77916 GRAM/driving circuitry stays fully powered during "screen off"; backlight is likely the dominant term but panel standby current is not zero | Low | `components/waveshare__esp32_s3_touch_lcd_1_85B/esp32_s3_touch_lcd_1_85B.c:1064-1068`; driver has `panel_st77916_disp_on_off` (DISPON/DISPOFF, 0x28/0x29) but no SLPIN/SLPOUT (0x10/0x11) support at all | Call `esp_lcd_panel_disp_on_off(panel, false)` alongside backlight-off in `screen_power_set_on(false)`; true SLPIN isn't implemented in this driver, so this is the best available step without patching the panel driver | Re-enabling requires `DISPON` + a short delay before the next flush; verify no visible glitch on wake | In progress — added `bsp_display_get_panel_handle()` accessor (`components/waveshare__esp32_s3_touch_lcd_1_85B/include/bsp/display.h`, `.../esp32_s3_touch_lcd_1_85B.c`) and wired `esp_lcd_panel_disp_on_off(panel, false/true)` into `screen_power_set_on()` (`main/app_clock/lvgl_screen_power_button.c`) alongside the existing backlight on/off, with a 20 ms settle delay after DISPON before backlight and brightness are restored. Looks correct on first pass; still needs on-device testing to confirm no stale-frame flash or tearing on wake before marking Done. |
| 4 | High | No intermediate "dim" stage — brightness jumps straight from 100% to 0% after 15 s idle | Medium — a dim step lets users glance at the clock without full brightness, extending perceived battery life without hurting the "check the time" use case | Low | `main/app_clock/lvgl_screen_power_button.c:12-14` (`IDLE_TIMEOUT_MS`) | Add a shorter first timeout that dims (e.g., 30%) via `bsp_display_brightness_set()`, keep the existing 15s→off as the second stage | None significant; matches the Active→Idle→Standby model | Not started |
| 5 | Medium | Audio codec (ES8311) and I2S opened unconditionally at boot and left open for the entire runtime | Medium — codec/I2S clocks likely stay active from boot even though no sound plays until an alarm/timer/chime | Moderate | `components/bsp_extra/src/bsp_board_extra.c:86-94` (`Audio_Play_Init()`), called unconditionally in `main/main.c:40` | Open the codec device just before `Audio_Play_Music()` and close it after playback finishes, instead of at boot | Adds latency to the first alarm/chime while codec re-opens; must verify `esp_codec_dev` supports safe re-open/close cycles | Not started |
| 6 | Medium | Five separate LVGL timers (clock 1s, battery 1s, timer-tile 500ms, idle-check 1s) keep firing continuously whenever the screen is on, regardless of which tile is actually visible, each doing an I2C transaction (RTC or fuel gauge) | Low-medium on its own (light sleep is already disabled whenever the screen is on) — but blocks introducing a "dimmed idle" mid-tier state cheaply | Low-moderate | `main/app_clock/lvgl_clock.c:188`, `main/app_clock/lvgl_device_status.c:53`, `main/app_clock/lvgl_timer.c:170`, `main/app_clock/lvgl_screen_power_button.c:122` | Slow these timers (e.g., 1s→5s) once a "dimmed idle" stage is added, and/or gate them on tile visibility via `lv_obj_has_flag(..., LV_OBJ_FLAG_HIDDEN)` | Battery/RTC labels would feel slightly less "live" while idle — acceptable trade-off; should not apply to the ringing alarm path | Not started |
| 7 | Low | Dev-only overhead left enabled: `CONFIG_ESP_LVGL_ADAPTER_ENABLE_FPS_STATS=y`, `CONFIG_LOG_DEFAULT_LEVEL_INFO`, debug compiler optimization, both UART and USB-Serial-JTAG consoles active | Low individually, but stacks up as continuous background work and no clean "production" build exists | Low | `sdkconfig` lines ~1636-1648, 1937, 2808, 899-908 | Add a `sdkconfig.release` (or menuconfig profile) that disables FPS stats, drops log level to WARN, and switches to `-Os`/release optimization for battery testing and shipping builds | None; purely a build-config split | Not started |

## Not flagged as high-impact (for reference, no action needed)

- SD card stays mounted permanently — correct trade-off since an alarm/timer/chime can fire at any moment and remounting has latency and correctness risk.
- IMU (QMI8658) is not initialized or polled anywhere — nothing to fix.
- Bluetooth is already disabled (`CONFIG_BT_ENABLED` unset).
